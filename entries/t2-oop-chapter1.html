Chapter 1 - Software and software engineering

Section 1.7
Activities common to software projects


Requirements and specification

The book defines three main factors we must understand in order to determine the best course of action for any given solution: The problem itself, the business environment, and the available technlogy. The general process includes following activities:

Domain analysis

This refers to what is known as Domain Knowledge This is the process in which we investigate common features between all the products and parts of the system, in order to discover potentially reusable designs. This is often learned from a user in the domain, an expert or specialist, rather than the software developers.

Defining the problem

One of the most important parts of understanding the problem, it's definining the Scope of the system. In simple words, this determines what is and what is NOT part of the features and functionalities the system must provide.

Requirements gathering

The process in which we obtain all the functional and non-functional requirements, everything the system is expected to do. This usually involves a middle man between the development team and the client. In Agile Methodologies, such as Scrum, a Product Owner is in charge of gathering the requeriments for the project.

Requirements analysis

Once we have gathered the requirements, we must organize them, and plan an action plan on how to execute the solution. This usually involves a discussion between the PO and the Development Team.

Requirements specification

One of the most important documents in every project. The specification will precisely point out every functionality or feature of the system. Both the "whats" (what is needed to solve the problem) and the "hows" (the desing and implementation). It's important to point out that this is bound to change, so requirement analysis should continue throughout the life of a software system.

Design

Design is the process of deciding how the requirements should be implemented using the available technology. This often involved the use of well defined and documented Design Patterns, among them, Creational, Sctructural, Behavioral and Concurrency Patterns. This design should always be defined to comply with several aspects of a system, to name a few:

  • Compatibility
  • Extensibility
  • Fault-tolerance
  • Maintainability
  • Reliability
  • Reusability
  • Performance
  • Portability
  • Scalability

Modeling

Modeling is the process of creating a representation of the domain or the software. Software modeling should address the entire software design including interfaces, interactions with other software, and all the software methods. The book mentions a few approaches:

  • Use Case modeling: Actions performed by users.
  • Structural modeling: Classes and objects.
  • Dynamic and behavioral modeling: States, activities and component interaction.
For object-oriented software, an object modeling language such as UML is often used.

Programming

So we have our specification, we've defined our design. It's time to execute. This means, translating those higher-level designs into particular programming languages. This requires knowledge of a particular programming language, in order to make decisions about the appropriate use of constructs, variable declarations, etc. which language to use varies from project to project. C#, Java, Node and Python, are common choices for Back-End and Middle-Tier development; JavaScript, TypeScript, Swift, and PHP are some popular options for Front-End.

Now, when it comes to the methodologies for programming, the team must decide on which one suits best their expertise and level of communication. The book mentions Extreme Programming (XP), though I would argue this methodology is a bit outdated and newer and more efective ones have emerged in the recent years. As mentioned before, Scrum is a very popular choice, along with Kanban, or even a combination of both. A very promising option gaining lots of popularity in recent years is SAFe.

Quality assurance

When it comes to delivering software, quality is key. It's not only about delivering a working product, but one that complies with all of the aspects previously mentioned in the design section. In order to achieve this, we must do several activities, usually "tests", to verify and validate that our system works as expected. To mention a few:

Functional tests:
  • Unit testing
  • Integration testing
  • System testing
  • Sanity testing
  • Smoke testing
  • Interface testing
  • Regression testing
  • Beta/Acceptance testing
  • Scalability
Non-functional testing types include:
  • Performance Testing
  • Load testing
  • Stress testing
  • Volume testing
  • Security testing
  • Reliability testing
  • Usability testing
  • Beta/Acceptance testing
  • Scalability

Deployment

Finally, once our system is complete, it's time to deploy it. This is, release it into a public environment where the user can access and use it. In recent years, the concepts of Continuous Integration and Continuous Delivery have been vital to the succesful deployment of software in an automated environment. Allowing the team to build, test, and release software with greater speed and frequency.
